Implement Timer Reset

I tried two implementations for this. Both are using the shared variable appraoch. The shared variable here is used to communicate inforamtion between threads.

The first one toggles a boolean. Most straightforward idea is to have a boolean variable work as the sential value to represent if some action needed to happen.

//bad do not do this
type Raft struct {
  //... some other fields
  bool timerExpired
  //... some other fields
}

// For election timer
for {
  rf.mu.Lock()
  rf.timerExpired = true
  rf.mu.Unlock()
  time.Sleep(electionTimeout)
}

// For heartbeat timer
for {
  rf.mu.Lock()
  rf.timerExpired = true
  rf.mu.Unlock()
  time.Sleep(heartbeatTimeout)
}

The main draw back of the boolean idea is that whenever timer is reset, we need to set the boolean value as false then wait for which ever loop to. It is very easy to forget to reset the timer.

A better strategy is “I will check continously; since my last check, has anthing happened yet?”. Or “if the system has seen evidence of activity from the legit leader(by receiving heart beat) since my last wakeup, I will not start a new election(because this is enough to convince me that the overall system has the leadership that it needs to operate properly)”.

// Good, do this.
func (rf *Raft) LeaderElection() {
	for !rf.killed() {
		startTime := time.Now()
		electionTimeout := 500 + (rand.Intn(500))
		time.Sleep(time.Duration(electionTimeout) * time.Millisecond)
		rf.mu.Lock()
		if rf.lastActive.Before(startTime) {
			if rf.state != Leader {
				go rf.KickoffElection()
			}
		}
		rf.mu.Unlock()
	}
	DPrintf("[%d] killed, stop leader election.", rf.me)
}

This implementation is better than the boolean toggle approach because the programmer no longer needs to remember to toggle back a boolean. Timestamp here automatically is adavancing. The responsibility of is shifted from the programmer to the system timer.

Tally Vote with the Conditional Variable Pattern

When implementing the leader election mechanism, any node as the Candidate role will first send out vote request in parallel and tally the vote.

This need of concurrency can be captured by the following pattern:

count, finished := 1, 1
cond = sync.NewCond(&mu)
for id := range peers {
  // Spawn many threads to run RPC in parallel; concurrenly do this is key for fault tolerance.
  go func task(server_id int) {
    lock()
    if voted {
      count++
    }
    finished++
    cond.Broadcast() // this should be called when count or finished is updated.
    unlock()
  }(id)
}


// Main thread
lock()
for !some_predicate(count, finished) {
  cond.Wait()
}
// Has lock and condition is true, process data
unlock()

Process reply.Term with rf.currentTerm(not the args.Term when msg are sent)

For raft to be correct, it is important for any node to ignore out dated messages. Therefore when RPC came back with reply from either RequestVote or AppendEntries, the leader’s decision should be based on rf.currentTerm to judge if a reply is worth process or should be discarded(because the rf.currentTerm has advanced from when it sent the RPC).

The information brought back by RPC is only useful if global Term has not advanced.

nextIndex and matchIndex usage

Remeber to record nextIndex and matchIndex for leader as well, because leader is also counted when decideing how to update commitIndex.

Prepare args in in critical sections, but send message out side of critical section.

Otherwise it will take deadlock. Example One: prepare AppendEntries and RequestVote args within lock. Then send msg with RPC. Then process the result within lock.

Debug Around Messages

For this assignment, all the logic changes that affects raft peer’s state are guarded between locks. All the message passing are not synchronized, the reason is RPC can be very slow and networking can be unreliable. Block and wait for the RPC return will cause deadlock. Because the handler is also guarded by lock.

Argument preparation is also done within lock.
